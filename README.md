# 社員体調管理・ご意見箱システム

本システムは、以下の2つの機能を統合した Web アプリケーションです。

1. **社員体調管理システム（Django）**  
   - 社員が毎日、体調（肉体）・気分（精神）を入力  
   - 管理者は統計を閲覧し、傾向を把握  
   - CSV＋DBを用いたデータ蓄積と pandas による簡易分析  

2. **ご意見箱システム（PHP）**  
   - 匿名／準匿名／署名の3形態で意見を投稿  
   - 投稿内容は CSV として保存  
   - Django から読み取り、分析可能

これらを **共通の Web インターフェース上からアクセス可能** とする構成です。

---

## 📌 目的

## 反省点・学び（Lessons Learned）

### 1. 設計フェーズに時間をかけすぎたことと、要件の曖昧さ
本プロジェクトでは設計フェーズに多くの時間を割いた一方で、  
設計書内に曖昧さや未確定な要素が残ったまま実装に入ってしまった点が課題だった。

事前に計画を立て、全体としては予定通りに進めることはできていたが、  
その曖昧さが要因となり、  
「この期間で実装すべきもの」と「将来拡張として切り分けるべきもの」の判断が弱くなった。

この経験から、**設計書を作る前段階で要件をどこまで固められているかが、設計と実装の質を大きく左右する**ことを学んだ。

---

### 2. UI / UX の検討が後手に回った点
当初は機能実装を優先し、UI は後から調整すればよいと考えていた。  
しかし実際には、体感的な使いやすさに欠ける部分が多く、  
**UI が直感的でなければ、そもそも使われない**という点を実感した。

機能要件だけでなく、画面構成や操作フローを含めた UI / UX 設計を  
初期段階から行う必要性を学んだ。

---

### 3. Django と PHP の二言語構成による負荷
学習目的で Django と PHP を組み合わせた構成に挑戦したが、  
結果として実装負荷が高くなり、それぞれの完成度を十分に高めきれなかった。

無理に複数言語を統合するよりも、  
**目的ごとにアプリケーションを分離して設計する判断も必要だった**と感じている。

---

### 4. セキュリティ設計の粒度が過剰だった点
健康情報を扱う前提から比較的厳密なセキュリティ設計を行ったが、  
その設計を正しく支えられるだけの基礎理解と実装力が十分ではなかった。

運用を想定する前に、まずは基本的なセキュリティ知識を確実に身につける必要があると感じた。

---

### 5. 実装スピードより「正しさ」を優先しすぎた点
設計を守ることを重視した結果、  
「まず動くものを作り、改善を重ねる」という開発のリズムを十分に活かせなかった。

正しさとスピードのバランスを取る重要性を学んだ。

---

### 6. それでも得られた最大の成果
これらの反省を通じて、自分の課題を具体的に言語化できたこと、  
そして初めて「設計から実装まで一貫して形にする」経験を積めたことは、大きな成果だったと考えている。

---

## 今後の改善方針・アプローチ（Future Approach）

### 1. 要件定義をより小さく切り、段階的に設計する
今後は最初から網羅的な設計書を作るのではなく、

- 最小限の要件（MVP）を定義  
- 実装 → 振り返り → 要件追加  

という段階的なアプローチを取ることで、  
設計と実装のズレを早期に発見・修正していく。

---

### 2. UI / UX を含めた画面設計を初期段階で行う
機能設計と並行して、

- 画面構成
- 操作フロー
- ユーザー視点での使いやすさ  

を意識した簡易的な画面設計（ワイヤーフレーム等）を作成し、  
UI / UX を前提とした実装を行う。

---

### 3. 技術選定は「学習目的」と「完成度」を分けて考える
今後は、

- 学習目的のプロジェクト  
- 完成度を重視するポートフォリオ  

を明確に分け、  
ポートフォリオでは無理に技術を組み合わせず、  
**一つの技術スタックで完成度を高めることを優先する**。

---

### 4. セキュリティは基礎を優先し、段階的に強化する
まずは、

- 認証・認可
- 基本的な入力バリデーション
- XSS / CSRF の基礎対策  

といった基本事項を確実に理解・実装し、  
その上で必要に応じて設計レベルを引き上げていく。

---

### 5. 「まず動くものを作る」開発プロセスを取り入れる
設計を意識しつつも、

1. 最低限動く状態を作る  
2. 問題点を洗い出す  
3. 改善・リファクタリングを行う  

というサイクルを意識し、  
実装経験を積み重ねるスピードを高めていく。
